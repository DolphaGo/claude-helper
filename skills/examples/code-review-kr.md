# 코드 리뷰 (한국어) Skill

## 목적
한국 개발 문화에 맞춘 체계적이고 건설적인 코드 리뷰를 제공합니다. 단순한 지적이 아닌 학습과 개선을 위한 피드백을 제공합니다.

## 사용 시점
- Pull Request 생성 후 리뷰 요청
- 신규 기능 구현 완료 후 검토
- 리팩토링 전후 비교
- 팀원 간 코드 품질 점검

## 실행 단계

### 1단계: 변경사항 파악
- Git diff로 수정된 파일 확인
- 커밋 메시지에서 의도 파악
- 영향 범위 분석

### 2단계: 체계적 검토

#### 2.1 구조 및 설계
- 아키텍처 패턴 준수 여부
- 모듈 간 의존성 적절성
- SOLID 원칙 준수

#### 2.2 코드 품질
- 가독성 (명명, 포맷팅)
- 복잡도 (중첩, 함수 길이)
- 중복 코드 존재 여부

#### 2.3 기능 및 로직
- 버그 가능성
- 엣지 케이스 처리
- 에러 핸들링

#### 2.4 성능
- 비효율적 알고리즘
- 불필요한 연산
- 메모리 누수 위험

#### 2.5 보안
- 입력 검증
- 인증/인가 체크
- 민감 정보 노출

#### 2.6 테스트
- 테스트 커버리지
- 테스트 품질
- 누락된 시나리오

### 3단계: 피드백 작성
각 항목별로 아래 형식으로 정리:
- 🔴 Critical: 반드시 수정 필요
- 🟡 Warning: 개선 권장
- 🟢 Suggestion: 선택적 개선
- 💡 Tip: 학습 포인트

### 4단계: 종합 평가
- 장점 (잘한 점)
- 개선점
- 다음 단계 제안

## 출력 형식
```
# 코드 리뷰 결과

## 📋 요약
- 파일: 3개 수정
- 라인: +120 / -45
- 전체 평가: ⭐⭐⭐⭐☆

---

## 🔍 상세 리뷰

### 파일: `src/auth.js`

🔴 **Critical: 보안 취약점**
- 라인 45: 비밀번호가 평문으로 로깅됨
- 권장: `logger.debug('Login attempt', { userId })` 로 수정

🟡 **Warning: 에러 처리 미흡**
- 라인 78-82: try-catch 블록에서 에러를 무시함
- 권장: 적절한 에러 로깅 및 사용자 피드백 추가

🟢 **Suggestion: 가독성 개선**
- 라인 102: 삼중 중첩 if문
- 권장: Early return 패턴으로 리팩토링

💡 **Tip: 성능 최적화**
- 라인 120: 루프 안에서 매번 정규식 생성
- 제안: 정규식을 상수로 선언하여 재사용

---

### 파일: `src/utils.js`

✅ **잘한 점**
- 함수명이 명확하고 직관적
- JSDoc 주석이 잘 작성됨
- 단위 테스트 추가

---

## 📊 종합 평가

### 👍 장점
1. 전반적으로 깔끔한 코드 구조
2. 타입 안정성 고려 (TypeScript)
3. 에지 케이스 일부 처리

### 🎯 개선점
1. 보안 관련 이슈 2건 해결 필요
2. 에러 핸들링 보완
3. 테스트 커버리지 향상 (현재 60% → 목표 80%)

### 📝 다음 단계
- [ ] Critical 이슈 수정 후 재검토 요청
- [ ] 보안 체크리스트 적용
- [ ] 통합 테스트 추가 고려
```

## 예제

### 입력
```
사용자: PR #123 코드 리뷰 해줘
```

### 실행 과정
```bash
# 1. PR 정보 가져오기
gh pr view 123 --json files,commits

# 2. Diff 확인
git diff main...feature-branch

# 3. 변경된 파일 분석
for file in $(git diff --name-only main...feature-branch); do
  # 파일별 상세 검토
done

# 4. 테스트 실행
npm test

# 5. 린트 체크
npm run lint
```

## 리뷰 체크리스트

### 필수 항목
- [ ] 버그나 로직 오류 없음
- [ ] 보안 취약점 없음
- [ ] 테스트 코드 포함
- [ ] 린트 규칙 통과
- [ ] 커밋 메시지 명확

### 권장 항목
- [ ] 코드 중복 최소화
- [ ] 함수 길이 적절 (<50줄)
- [ ] 복잡도 관리 (Cyclomatic < 10)
- [ ] 주석 적절히 사용
- [ ] 에러 처리 완전

## 주의사항
- **존중하는 태도**: 비판이 아닌 건설적 피드백
- **구체성**: "별로다" ❌ → "여기서 X를 Y로 바꾸면 더 좋습니다" ✅
- **우선순위**: Critical > Warning > Suggestion 순으로
- **학습 기회**: 단순 지적보다 이유와 대안 제시

## 한국 문화 고려사항
- 직접적 표현보다 완곡한 제안
- "잘못됐다" → "이렇게 하면 더 좋을 것 같아요"
- 좋은 점 먼저 언급 후 개선점 제시
- 선배/후배 관계 고려한 톤

## 관련 도구
- `Bash`: git diff, lint, test 실행
- `Grep`: 패턴 기반 이슈 검색 (TODO, FIXME, console.log)
- `Read`: 파일 내용 상세 분석

## 자동화 팁

### 공통 이슈 자동 탐지
```bash
# console.log 남아있는지 확인
grep -r "console\.log" src/

# TODO 카운트
grep -r "TODO" src/ | wc -l

# 긴 함수 찾기
eslint --rule 'max-lines-per-function: [error, 50]'
```

### pre-commit hook 연동
```bash
# .git/hooks/pre-commit
npm run lint
npm test
```
